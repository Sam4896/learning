DSPy — The AI Compiler

Stop writing prompts by hand. DSPy lets you declare what you want from an AI and automatically finds the best prompt to get it.

Python library·Works with Claude, GPT, Gemini·No GPU needed

The Problem DSPy Solves

Analogy first Imagine writing a web application where you had to manually write every CPU instruction. That would be insane — you use a compiler. Building LLM apps without DSPy is the same: you are manually "compiling" prompts by hand, and every time you change the AI model, they break.

Without DSPy — Hand-Written Prompts

"You are an expert researcher.
Given this question: {question}
And context: {context}
Answer precisely. Use Vancouver
citations. Max 300 words. Do
not use first person. Always..."

Fragile. Hard-coded. Breaks when you switch models. Impossible to optimize systematically. Grows unmanageable.

With DSPy — Declarative

class Answer(dspy.Signature):
    """Answer the question
    with source citations."""
    question: str = InputField()
    context:  str = InputField()
    answer:   str = OutputField()

Declarative. Model-agnostic. The actual prompt text is generated and optimized automatically by DSPy.

The PyTorch Analogy (from the original Stanford paper) In PyTorch, you write a forward pass in Python and the framework figures out the backward pass. In DSPy, you write a pipeline in Python and the framework figures out the prompts. Same principle: you specify the structure; the system finds the parameters.

The Three Core Abstractions

Every DSPy program is built from exactly three things. Click each block to expand a detailed explanation:

📋 Signature — The Contract

A Signature declares what inputs go in and what outputs should come out of an LLM call — without specifying how to get there. Think of it as a typed function signature, but for AI tasks.

String form (quickest to write)

# Minimal: "inputs -> outputs"
qa = dspy.Predict("question -> answer")
result = qa(question="What is DSPy?")
print(result.answer)

# With type annotation
clf = dspy.Predict("text -> label: bool")

Class form (full control)

class SummarizeCode(dspy.Signature):
    """Summarize what this code does
    in plain English."""
    code: str = dspy.InputField()
    summary: str = dspy.OutputField(
        desc="2-3 sentence summary"
    )
How the optimizer sees it The class docstring becomes the task instruction given to the LLM. The desc on each field guides what the LLM puts there. The optimizer can rewrite both — that is how it improves prompts. Every tunable part is accessible.

Internally, a Signature is a Pydantic BaseModel. The optimizer has structured read/write access to the instruction text and all field descriptions.

⚙️ Module — The Strategy

A Module wraps a Signature and adds a prompting strategy. Two modules can share the exact same Signature but prompt the LLM in completely different ways. The Signature is the what; the Module is the how.

ModuleStrategyUse when…
PredictDirect answer, no extrasSimple fast tasks, classification
ChainOfThoughtReason step-by-step, then answerAny reasoning or multi-hop problem
ReActReason + call tools in a loopAgentic tasks requiring tool use
ProgramOfThoughtWrite & execute Python codeMath, computation, data analysis
MultiChainComparisonGenerate N candidate answers, compareHigh-stakes decisions
RetryRe-call with error feedback if assertion failsWhen output must satisfy constraints
Same contract, different strategy You can swap Predict for ChainOfThought at any time without touching the Signature. The "what" is stable; only the "how" changes. This separation is what makes DSPy programs easy to improve incrementally. 
# Same Signature, three different strategies:
direct  = dspy.Predict("question -> answer")
thinker = dspy.ChainOfThought("question -> answer")
agent   = dspy.ReAct("question -> answer", tools=[search_web])

🔧 Optimizer — The Compiler

The Optimizer takes your program + training examples + a scoring function and automatically finds better prompts. No gradients, no GPU — pure discrete text search.

Bootstrap — Run your unoptimized program on training examples. Record every LLM call, every input, every output (the "trace").
Generate Candidates — Extract successful traces as few-shot examples. Ask a separate LLM to propose improved instruction variants for the docstring.
Evaluate — Test each instruction+demo combination on a held-out validation set. Score with your metric function (returns 0.0–1.0).
Save Winner — Serialize the best combination as a .json file. Load it later in milliseconds — no re-optimization ever needed again.
The compiled .json is just text It contains the winning instruction string and winning few-shot examples. No model weights, no GPU. Extract the text and use it anywhere — even outside DSPy.

Building Your First DSPy Program

A complete end-to-end example with every step annotated:

import dspy

# ── Step 1: Connect to an LLM ─────────────────────────────────────────
lm = dspy.LM("anthropic/claude-sonnet-4-20250514")
dspy.configure(lm=lm)
# DSPy also supports: "openai/gpt-4o", "google/gemini-pro", any litellm model

# ── Step 2: Declare WHAT you want (Signature) ─────────────────────────
class AnswerWithReasoning(dspy.Signature):
    """Answer the question based on the context.
    Cite sources using [1], [2] notation."""
    question: str = dspy.InputField(desc="The user's question")
    context:  str = dspy.InputField(desc="Background information")
    answer:   str = dspy.OutputField(desc="Precise answer with inline citations")

# ── Step 3: Define HOW to execute (Module) ────────────────────────────
class SmartQA(dspy.Module):
    def __init__(self):
        # ChainOfThought adds a hidden "rationale" reasoning step
        self.answer = dspy.ChainOfThought(AnswerWithReasoning)

    def forward(self, question, context):
        return self.answer(question=question, context=context)

# ── Step 4: Run it immediately (no optimization required to just run) ──
qa = SmartQA()
result = qa(
    question="What is gradient descent?",
    context="Gradient descent minimizes a function by..."
)
print(result.rationale)  # step-by-step reasoning (added by ChainOfThought)
print(result.answer)     # the final answer

# ── Step 5: Optimize it against your own examples ─────────────────────
def my_metric(example, prediction, trace=None):
    # Your scoring function: return float 0.0 to 1.0
    return score_answer_quality(prediction.answer, example.expected_answer)

# MIPROv2 is a good general-purpose optimizer
optimizer = dspy.MIPROv2(metric=my_metric, auto="medium")
compiled  = optimizer.compile(qa, trainset=your_training_data)
compiled.save("optimized_qa.json")    # save once, reuse forever

# ── Later: load without re-optimizing ─────────────────────────────────
qa2 = SmartQA()
qa2.load("optimized_qa.json")          # instant, no API calls needed

Composing a Multi-Step Pipeline

The real power is chaining modules in plain Python. Any if, for, or function call is valid — DSPy does not impose a graph structure.

class ResearchPipeline(dspy.Module):
    def __init__(self):
        # ReAct uses tools (here: a search function)
        self.find_papers  = dspy.ReAct("topic -> papers: list", tools=[search_arxiv])
        # ChainOfThought for reasoning over retrieved results
        self.synthesize   = dspy.ChainOfThought("topic, papers -> synthesis, gaps")
        # Predict for final structured output
        self.write_review = dspy.Predict("topic, synthesis, gaps -> review: str")

    def forward(self, topic):
        # Each call is a separate optimizable LLM interaction
        papers    = self.find_papers(topic=topic)
        synthesis = self.synthesize(topic=topic, papers=papers.papers)
        review    = self.write_review(
            topic=topic,
            synthesis=synthesis.synthesis,
            gaps=synthesis.gaps
        )
        return review

# The optimizer improves ALL three sub-modules simultaneously
optimizer = dspy.GEPA(metric=review_quality_metric)   # ← see the GEPA guide!
compiled  = optimizer.compile(ResearchPipeline(), trainset=data)

Integrating DSPy with LangGraph

DSPy and LangGraph solve different but complementary problems. LangGraph owns the workflow (which step runs next, how state flows). DSPy owns the LLM call quality within each step.

The pattern: compile DSPy modules offline, then call them from inside LangGraph nodes.

from langgraph.graph import StateGraph
from typing import TypedDict
import dspy

# Pre-compile DSPy programs offline (not during live execution)
lit_reviewer = LitReviewPipeline()
lit_reviewer.load("compiled_lit_reviewer.json")

class PaperState(TypedDict):
    topic: str; literature: str; methods: str

# Each LangGraph node wraps a DSPy call
def lit_review_node(state: PaperState) -> PaperState:
    result = lit_reviewer(topic=state["topic"])
    return {"literature": result.review_paragraph}

workflow = StateGraph(PaperState)
workflow.add_node("literature", lit_review_node)
workflow.set_entry_point("literature")
app = workflow.compile()

# The workflow structure is stable; LLM quality is separately optimizable

When to choose GEPA over MIPROv2 If your program has multiple steps, uses tools, or fails in ways that are hard to capture with a single score — choose GEPA. It can read the full execution trace and diagnose where a failure occurred. See the GEPA guide for details.

DSPy Traces — The Bridge to GEPA

A trace is the complete execution record of a program run: every LLM call, every input, every output, and every tool invocation. Traces are the data structure that connects DSPy to GEPA.

# Traces are captured automatically during compilation.
# You can also inspect them manually:
with dspy.context(trace=[]):
    result = compiled_program(topic="gradient descent")
    trace  = dspy.settings.trace  # full execution record

# A trace for a 3-module pipeline looks like:
[
  {"module": "find_papers",  "input": {"topic": "..."},
   "output": {"papers": [...]}, "tool_calls": [...]},
  {"module": "synthesize",   "input": {"topic": "...", "papers": [...]},
   "output": {"synthesis": "...", "gaps": [...]}},
  {"module": "write_review", "input": {"..."},
   "output": {"review": "..."}, "score": 0.82}
]

GEPA consumes these traces. Instead of seeing only the final score, it reads the complete trace and asks a reflection LLM: "Why did this fail? What step went wrong?" — then generates a targeted fix.

Check Your Understanding

Q: You want a system that searches the web for papers and summarizes them. Which module handles the search step?

Correct! ReAct (Reason + Act) is the module built for tool-using agents. It lets the AI call your search function and reason about results in a loop. The others cannot call external tools.

Q: The Optimizer produces a .json file. What does it contain?

Correct! Just text — the winning instruction string + selected few-shot demonstrations. No GPU, no model weights. Load it in milliseconds and use it without ever re-running the optimizer.

Q: What is the relationship between a Signature and a Module?

Correct! Signature = WHAT (typed inputs/outputs). Module = HOW (strategy: Predict, ChainOfThought, ReAct, etc.). You can swap the Module without touching the Signature. This clean separation is what makes DSPy programs modular and optimizable.
