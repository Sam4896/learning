What Is Hermes Self-Evolution?

Analogy Imagine if Claude Code could look back at conversations where it made mistakes, figure out which of its own instruction files caused those mistakes, and then rewrite those files to be better — all automatically, overnight, for a few dollars. That is what Hermes self-evolution does.

Hermes is an AI coding agent (similar to Claude Code). Like Claude Code, it runs from skill files — markdown documents that tell it how to do specific tasks: how to review a GitHub PR, how to write LaTeX, how to run a test suite. The self-evolution repo uses DSPy + GEPA to optimize those skill files as text-based optimization targets.

What Gets Optimized

  • Skill files (~/.hermes/skills/*.md)
  • Tool descriptions (planned Phase 2)
  • System prompt sections (planned Phase 3)
  • Tool Python implementation (planned Phase 4)

What Drives Improvement

  • Synthetic benchmark task generation
  • Your real Claude Code / Copilot sessions
  • Explicit success metrics per skill
  • GEPA's reflective evolution loop

The 5-Phase Evolution Roadmap

Phase 1 Skill File Evolution ✓ Implemented

Target: ~/.hermes/skills/*.md — the procedure files that tell Hermes how to do specific tasks.

Why first: Skill files are self-contained, have clear purpose, and are easy to define evaluation metrics for. Failures are local to the skill.

Phase 2 Tool Description Evolution Planned

Target: The short description string attached to each tool that tells the AI what it does, when to use it, and what arguments to provide.

Why it matters: Tool descriptions affect every single tool call across every session. A 10% improvement in tool call accuracy compounds into a large gain over many interactions.

Before: "search_papers: Search for academic papers."

After GEPA: "search_papers: Search Semantic Scholar by topic, author, or keyword. Returns titles, abstracts, DOIs. Use for prior work checks. Do NOT use for full PDF text (use fetch_pdf instead)."

Phase 3 System Prompt Evolution Planned

Target: The system prompt sections that define Hermes's overall behavior, persona, and priorities.

Why harder: System prompts affect everything. A bad mutation cascades across all tasks and is hard to diagnose. Requires fine-grained section-by-section optimization with targeted test suites to avoid unintended regressions.

Approach: Divide the system prompt into named logical sections (persona, formatting, tool use policy, safety constraints). Evolve each section independently against a focused test suite. Guard against cross-section interference.

Phase 4 Code Evolution (Darwinian Evolver) Planned

Target: The actual Python implementation of tools — not just their descriptions, but their runtime behavior.

Different engine: Uses imbue-ai/darwinian_evolver instead of GEPA because code has hard correctness constraints: it must compile, execute without crashing, and pass all tests. GEPA's text-mutation approach too easily produces broken Python.

How it works: Treats each candidate as a git commit. Mutations are LLM-generated code edits. Selection is based on test pass rate + performance benchmarks. Crossover combines code sections from two high-performing commits. Like genetic algorithms but at the level of git history.

Phase 5 Continuous Improvement Loop Planned

Target: A fully automated pipeline running all phases on a schedule, reading your session history, identifying skill failures, and deploying improvements without manual intervention.

The vision: A weekly cron job that reads your real conversations, finds which skill files contributed to poor outcomes, runs GEPA on those specific skills, passes all guardrail checks, and deploys the winners automatically. Your agent gets marginally better every week, at the cost of a cup of coffee per run.

How Phase 1 Skill Evolution Works (Step by Step)

Input: A skill file

e.g. github-code-review.md — contains: name, purpose, step-by-step procedure, edge cases, and examples.

Build Evaluation Dataset

Synthetic mode: An LLM generates 20 realistic tasks that would trigger this skill, plus expected outcomes. Fast, always available.
SessionDB mode: Reads ~/.claude/history.jsonl and Copilot sessions. Filters for conversations where this skill was relevant (2-stage: keyword filter → LLM-as-judge).

Define the Metric

Run Hermes with the current skill file on each test case. Score the result. Generate rich feedback: "The skill failed because it didn't specify to check test coverage before approving the PR."

GEPA Evolution Loop (10–15 iterations)

The skill file text itself is the "prompt" being optimized. GEPA runs: evaluate → reflect → mutate → select. The Pareto front balances skill quality against skill file size.

Guardrail Checks (mandatory)

Every variant must pass: pytest tests/ -q (100%), size ≤ 15KB, caching compatibility, and a semantic preservation check — an LLM judge asks "Does this evolved version still do what the original claimed?" to prevent GEPA from gaming the metric by changing the skill's purpose.

Output: Deploy or PR

Either opens a PR against the Hermes repo for human review (recommended), or you copy directly to ~/.hermes/skills/ for immediate local use. The PR model ensures a human checks the change before it affects live behavior.

SessionDB — Learning From Your Real Work

Key Insight Every conversation where Claude Code didn't quite do what you wanted is a signal that a skill or instruction could be better. SessionDB harvests that signal automatically, turning your failure history into a curated evaluation set.

SessionDB reads from these files on your machine:

~/.claude/history.jsonl           # Your Claude Code sessions
~/.hermes/sessions/               # Hermes's own session store
~/.copilot/session-state/*/       # GitHub Copilot sessions

It then filters with a two-stage pipeline to find conversations relevant to the skill being evolved:

Stage 1: Keyword Filter (Fast)

Scan each conversation for keywords related to the skill. Runs in milliseconds. Discards obvious mismatches cheaply before touching the LLM.

Example: skip conversations about Python dependencies when evolving a "latex-writing" skill.

Stage 2: LLM-as-Judge (Accurate)

For conversations passing Stage 1: "Would the latex-writing skill have been relevant to this conversation? Score 0–1." Built as a DSPy module — so it too can be optimized.

Keep conversations with score > threshold. These become your personalized eval set.

Guardrails & Safety Constraints

Every evolved variant must pass hard constraints before it can be deployed:

ConstraintCheckWhy
Full test suitepytest tests/ -q passes 100%Evolved variants must not break existing functionality
Size limitSkills ≤ 15KB; tool descriptions ≤ 500 charsPrevent context-bloating over-specified skills
Caching compatibilityNo mid-conversation state changesHermes caches skills; a skill that changes meaning mid-session breaks the cache
Semantic preservationLLM judge: "Does this still do what the original claimed?"Prevents GEPA from gaming the metric by changing the skill's purpose
Human reviewAll changes go through PR, never direct commit to shared repoFinal human-in-the-loop gate before it affects the live agent
The semantic preservation check is the most important Without it, GEPA could "cheat" by evolving a code review skill into a rubber-stamp skill that always says "LGTM" — which scores perfectly on easy benchmarks but is obviously wrong. The LLM judge catches this by comparing the evolved skill's intent to the original.

The 6 Mental Models to Carry Forward

1. Prompts are compiled programs

A prompt is not just text — it encodes your intent the same way code does. Like all programs, it can be compiled (DSPy), tested (metric function), and optimized (GEPA). Stop writing prompts by hand; write evaluation functions instead.

2. Scalars are lossy

Reducing a failure to a score of "0.4" throws away the diagnosis. GEPA preserves the full execution trace so a reflection LLM can understand exactly what went wrong and propose a precise fix.

3. Targeted beats random

Random prompt variations are inefficient. A targeted mutation — "fix the citation format reference in step 2" — converges 35x faster than RL-style random perturbation because it is guided by understanding, not probability.

4. Skills are textual programs

A markdown skill file has implicit inputs, outputs, and measurable behavior. It can be evaluated on benchmark tasks and improved via GEPA. This insight is the foundation of Hermes self-evolution.

5. Your history is training data

Every past conversation where the AI underperformed is a training signal. SessionDB extracts this signal automatically from ~/.claude/history.jsonl and turns it into a curated evaluation set for the skills that caused those failures.

6. No GPU, API cost only

Everything here works with LLM APIs. No model fine-tuning, no GPU, no custom infrastructure. A full GEPA optimization run costs $2–10 and takes minutes to a few hours. The barrier to entry is extremely low.

Practical Roadmap

Step 1 Shift mindset: write evaluation functions, not prompts

Your CLAUDE.md and custom skill files are structured prompts with implicit inputs and outputs. Instead of editing them by hand, define what good looks like as a scoring function and let DSPy + GEPA find the best phrasing.

Concretely: write 10–20 examples of tasks you want done correctly, plus a function that scores the output. That is your entire contribution. The optimization is automatic.

Step 2 Optimize your MCP tool descriptions with GEPA

Every MCP tool in your ~/.claude/mcp.json has a description string. GEPA can evolve these to make Claude call them more accurately.

from gepa.adapters.mcp import MCPAdapter

adapter = MCPAdapter(
    server_config={"type": "stdio", "command": "my-server"},
    tools_to_optimize=["search_code", "edit_file", "run_tests"],
    feedback_fn=lambda gold, pred: (
        score_tool_usage(pred),
        f"Tool called with wrong argument: {pred.args}"
    )
)
evolved = adapter.run(num_iterations=10)
Step 3 Build a DSPy program for a complex repeated task

If you have a multi-step task you repeat regularly (e.g., "analyze paper → extract claims → write methods section → format citations"), encode it as a DSPy pipeline. Compile it once against your own examples. Use the compiled .json as the instruction set forever.

class MyResearchWorkflow(dspy.Module):
    def __init__(self):
        self.extract = dspy.ChainOfThought("paper_text -> claims: list")
        self.write   = dspy.Predict("claims, equations -> methods_tex: str")

The Full Workflow in Three Sentences

DSPy lets you declare AI tasks as typed Signatures and compose them into pipelines using plain Python, separating the what (your intent) from the how (the prompt).

GEPA takes any text-based parameter, runs it on test cases, uses an LLM to read the failure traces and diagnose what went wrong in plain English, then generates targeted mutations — 35x cheaper than RL and human-readable throughout.

Hermes self-evolution applies this same loop to an AI agent's own skill files, reading real session history to identify failures and automatically improving the specific skills that caused them — all at API cost, all without a GPU, all with a human reviewing the final PR before it ships.

Final Quiz — All Three Topics

Q: GEPA is 35x more sample-efficient than RL. The core reason is:

Exactly right! RL says "score = 0.4." GEPA says "step 3 failed because the citation format was wrong — expected [1] but got (Smith, 2024)." That specificity lets GEPA propose a precise fix in 1–3 iterations that RL would need hundreds of random attempts to stumble upon.

Q: In DSPy, a Signature answers which question about an LLM call?

Correct! Signature = WHAT (typed inputs/outputs). Module = HOW (Predict, ChainOfThought, ReAct). Optimizer = HOW GOOD + how to improve it. This three-layer separation is what makes DSPy programs composable, swappable, and automatically optimizable.

Q: Hermes self-evolution uses DSPy + GEPA on its own skill files. What makes this possible?

Correct! The key insight: a markdown skill file can be run on benchmark tasks and scored, just like a prompt. That makes it a valid GEPA target. The only requirement for GEPA to optimize something is: (1) it has a textual parameter, and (2) you can evaluate it and return a (score, feedback) pair.

Q: You write a GEPA feedback function that returns (0.7, "word count 420 exceeds target 300"). How does GEPA use the string part?

Correct! The feedback string is the primary input to the Reflect step. The reflection LLM reads "word count 420 exceeds target 300," understands the problem, and proposes a mutation like "Add 'Keep your answer under 300 words' to the instruction." Without the string, the LLM would only see 0.7 and have no idea what to fix.

Q: Hermes self-evolution uses a "semantic preservation check" before deploying any evolved skill. What is it checking for?

Correct! Without this check, GEPA could "cheat" by, for example, evolving a code review skill into one that always says "LGTM" — which scores 100% on naive benchmarks but is obviously wrong. The LLM judge compares the evolved version's intent to the original and flags drift before deployment.
