GEPA — The Evolutionary Optimizer

GEPA uses an AI to read failure logs, understand why something went wrong, and propose a targeted fix. Then repeats. 35x cheaper than Reinforcement Learning.

Genetic-Pareto algorithm·No GPU·Human-readable logs·$2–10 per run

What Is GEPA?

GEPA stands for Genetic-Pareto. It is a prompt and system optimizer introduced in the paper "GEPA: Reflective Prompt Evolution Can Outperform Reinforcement Learning" (arXiv:2507.19457, ICLR 2026 Oral).

It was released simultaneously as a standalone library (pip install gepa) and as a native DSPy optimizer (dspy.GEPA). The core idea: instead of optimizing a prompt by computing gradients or doing random search, you use an LLM to read the failure trace, diagnose the cause in natural language, and propose a targeted fix.

Analogy Imagine teaching someone to bake by only ever saying "that scored 0.4" after each attempt — that is Reinforcement Learning. GEPA instead says: "The sponge was too dense because you didn't fold the egg whites gently in step 3. Fix that." Now the learner knows exactly what to change.

Why Not Reinforcement Learning?

Reinforcement Learning (RL)

  • Returns a scalar reward only: "0.43"
  • Needs 5,000–25,000+ evaluations to converge
  • Requires GPU + direct model weight access
  • Cannot explain why something failed
  • Takes hours to days
  • Expensive infrastructure required

GEPA

  • Reads full execution traces: "step 3 failed because…"
  • Needs only 100–500 evaluations
  • API-only, no GPU required
  • Human-readable improvement logs
  • Takes minutes to a few hours
  • $2–10 total cost per optimization run
The key difference RL reduces a complex failure to a single number. GEPA preserves the full execution context — error messages, reasoning traces, intermediate outputs, tool call results — and passes all of it to a reflection LLM that understands natural language. Language is a richer optimization signal than a scalar.

📊 Evaluate — Capture Everything

GEPA runs each candidate prompt on your test examples and captures three things per run — not just a score:

  • Score (0.0–1.0) — how well the output matched expectations
  • Execution trace — every LLM call, tool invocation, reasoning step, and intermediate output
  • Feedback string — your natural-language description of what went right or wrong

This richness — especially the trace and feedback string — is what separates GEPA from RL, which only ever sees the scalar score.

🔍 Reflect — Diagnose in Language

A Reflection LLM reads the execution traces from failed runs and produces a natural-language diagnosis:

# What GEPA sends to the Reflection LLM:
"""
Candidate prompt: "Answer the question concisely."
Test case: "What is gradient descent?"
Expected:  Clear explanation with the update rule equation.
Actual:    "Gradient descent finds the minimum of a function."
Feedback:  "Too vague. Missing the update rule: theta -= lr * grad(L)"
Trace:     [single direct call, no reasoning, no math]
Why did this fail? What specifically needs to change?
"""

The reflection LLM responds: "The prompt does not request mathematical detail. It needs to specify that technical answers should include the formal update rule."

✎️ Mutate — Targeted Fixes

Based on the reflection diagnosis, GEPA generates N new prompt candidates that address the identified issue. These mutations are semantic and targeted — not random perturbations.

# The mutation step in action:
original:   "Answer the question concisely."
# Mutation 1: addresses the missing math requirement
mutation_1: "Answer the question. For algorithm or math questions,
             always include the formal update rule or equation."
# Mutation 2: more explicit structure
mutation_2: "Provide a precise technical answer structured as:
             (1) intuitive explanation (2) mathematical formulation
             using standard notation (3) when/why to use this."
Evolutionary Memory: Each mutation also inherits lessons from all ancestral candidates — improvements compound across generations.

🏆 Select — The Pareto Front

New candidates are evaluated, then added to the Pareto Front if they are "non-dominated." A candidate is non-dominated when no other candidate beats it on all objectives simultaneously.

Why Pareto and not just "pick the best score"? Because you often care about multiple objectives: quality and cost and latency. Pareto keeps all candidates that represent a valid trade-off point.

The dominated point (gray) is beaten by "Balanced" on both quality and cost — so it is discarded. The three orange points each win on at least one dimension.

The Feedback Function — Your Most Important Design Choice

The Golden Rule of GEPA The richer and more specific your feedback string, the better the mutations GEPA proposes. A score of "0.6" produces vague mutations. A string like "citation format was wrong: expected [1] Smith (2024) but got (Smith, 2024)" produces a precise, surgical fix in the very next iteration. 
def my_feedback_function(gold, pred, trace=None) -> tuple[float, str]:
    """Returns: (score: float 0.0-1.0,  feedback: str)
    'feedback' is natural language. It describes WHAT went wrong.
    This is exactly what GEPA's Reflection LLM reads to propose
    the next mutation. Be specific!"""
    score  = 0.0
    issues = []
    # ── Check 1: citation format ─────────────────────────────────
    if pred.citation_format_correct():
        score += 0.5
    else:
        issues.append(
            f"Citation format wrong. Expected [1] Smith (2024), "
            f"got '{pred.citations[:40]}'. Use numbered Vancouver style."
        )
    # ── Check 2: word count in target range ──────────────────────
    wc = pred.word_count()
    if 200 <= wc <= 300:
        score += 0.3
    else:
        issues.append(f"Word count {wc} outside target 200-300 words.")
    # ── Check 3: academic register (passive voice) ───────────────
    if pred.passive_voice_ratio() > 0.6:
        score += 0.2
    else:
        issues.append("Should use passive voice throughout (academic register).")
    feedback = " | ".join(issues) if issues else "All checks passed."
    return score, feedback

Two Ways to Use GEPA

Option A — dspy.GEPA

Drop-in replacement for any DSPy optimizer. Best when you already use DSPy modules. Optimizes signatures, instructions, and few-shot examples all at once.

# Same API as MIPROv2 or BootstrapFewShot
optimizer = dspy.GEPA(
    metric=my_feedback_fn,  # returns (score, feedback_str)
    num_iterations=10,
    population_size=5,
    num_mutations=3,
)
compiled = optimizer.compile(
    my_dspy_program,
    trainset=data
)
compiled.save("evolved.json")

Option B — Standalone gepa

Optimize any text parameter — system prompts, tool descriptions, config strings, non-DSPy pipelines. Maximum flexibility.

from gepa import GEPA, TextComponent
# Define what you want to optimize
component = TextComponent(
    text="You are a helpful assistant.",
    name="system_prompt"
)
# Run evolution
g = GEPA(components=[component], metric=my_fn)
best = g.run(test_cases=[...])
print(best.text)  # the evolved system prompt

When Does GEPA Stop?


GEPA is budget-based. You configure when it stops using composable StopCondition objects:


from gepa import GEPA, CompositeStopper
from gepa.stop import MaxMetricCallsStopper, NoImprovementStopper, ScoreThresholdStopper

stopper = CompositeStopper([
    MaxMetricCallsStopper(max_calls=200),     # hard budget cap
    NoImprovementStopper(patience=3),         # stop if 3 iters with no gain
    ScoreThresholdStopper(threshold=0.95),    # stop if target quality reached
])
# TimeoutStopCondition(seconds=3600)  ← wall-clock limit
# FileStopper("stop.flag")            ← touch a file to stop manually
# SignalStopper(signal.SIGTERM)       ← OS signal

Why "Report Card" Optimization Fails

    

    
Before GEPA, the usual way to optimize an LLM was Reinforcement Learning (RL). Let's understand why RL struggles with text, using an analogy you already know.

    

      
Analogy — The Report Card

      Imagine you write an essay. Your teacher only tells you: "Score: 62/100."

      No comments. No feedback. Just a number.


      How do you improve? You can't. You'd have to randomly rewrite the essay and hope the number goes up.
      That's what RL does to an LLM.
    

    

      
RL in 3 steps — and where it breaks

      

        
1. Try a prompt

        →
        
2. Get a score (reward)

        →
        
3. Nudge the parameters

      

      
The problem: text is discrete. If your prompt has 50 tokens, there are roughly 50,00050 possible prompts. You can't compute a gradient through that. Small changes to a prompt can make the score jump wildly — or do nothing.

      

        
New term — Sparse Reward

        RL needs a reward signal on every step to guide learning. But for prompts, we only know the final score. That's a sparse reward: almost always zero, occasionally one. Learning from this is like navigating a dark maze that only lights up when you reach the exit.
      

    

    

      
What GEPA does instead — "Here's exactly what went wrong"

      
Instead of a number, GEPA gives the LLM a full diagnosis:

      
-- RL reward --
score: 0.62

-- GEPA feedback (called "Actionable Side Information") --
failed_cases: [
  { input: "What is 8 × 7?", got: "54", expected: "56" },
  { input: "Translate 'hello'", got: "hola", expected: "bonjour" }
]
trace: "model skips showing work for multiplication"
diagnosis: "prompt does not instruct step-by-step computation"

      
Now the optimizer knows exactly what to fix. No guessing. This is called Actionable Side Information (ASI).

    

  


  


  
  

    

      
2

      
Multi-Objective Optimization

    

    
Here's the second problem with RL: it optimizes one number. But real-world prompts must balance multiple goals at once.

    

      
Real-world example

      You want a customer-service bot that is:

      accurate concise polite


      These goals conflict: being more verbose often improves accuracy but hurts conciseness.
      Collapsing them into one score (accuracy×0.4 + conciseness×0.3 + politeness×0.3) forces you
      to pick weights arbitrarily. What if you don't know the right weights?
    

    
Formal definition

    
Let's write this mathematically. Suppose you have K objectives, and a prompt candidate called θ. The score of θ on objective i is:

    

      score function
      Fi(θ) = 𝔼(x,y) ~ 𝒟 [Mi(Pθ(x), y)]
    

    

      
SymbolMeaningExample

      
Fi(θ)How well prompt θ does on objective iF₁ = accuracy on test set

      
θThe prompt candidate (instructions + few-shot examples)"Always think step by step. Here are 3 examples..."

      
𝔼(x,y)~𝒟Average over the dataset 𝒟Mean over 100 math questions

      
Pθ(x)LLM's answer to x using prompt θGPT-4's output given the prompt

      
MiMetric function iM₁ = exact match, M₂ = output length

    

    
So you have a vector of scores: F(θ) = [F₁(θ), F₂(θ), ..., FK(θ)].

    
The problem: which θ is "best" when you have K numbers instead of one? This is where Pareto dominance comes in.


    

      
Pareto Dominance — the "strictly better" idea

      
We say prompt θ dominates θ' (written θ ≻ θ') if:

      

        Pareto dominance
        θ ≻ θ'  ⟺  ∀i: Fi(θ) ≥ Fi(θ')  and  ∃j: Fj(θ) > Fj(θ')
      

      

        
Plain English

        θ dominates θ' means: θ is at least as good on every metric, and strictly better on at least one metric. In other words — θ is better in some way and worse in no way.
      

      
A prompt that nobody dominates is called Pareto-optimal. The set of all Pareto-optimal prompts is the Pareto Front.

The GEPA Algorithm — Step by Step

    

    
GEPA stands for Genetic-Pareto Optimizer. It's an evolutionary algorithm: it starts with a population of prompt candidates and iteratively breeds better ones. Click through each step below.

Reflection + Mutation: the Math

    

    
The heart of GEPA is a two-stage transformation: Reflect (diagnose) then Mutate (fix). Let's make this precise.

    

      
Analogy — The Debugging Loop

      You have buggy code. A senior developer reads the error messages and traces (Reflect)
      and writes a diagnosis: "The bug is in the loop index." Then they edit the specific line
      (Mutate). They don't randomly rewrite the whole file — they make targeted changes.
    

    
Step 1 — Reflection

    
Given a prompt θ that failed on a minibatch 𝒮, the Reflect operator calls an LLM to produce a natural-language diagnosis:

    

      Reflect
      d = LMreflect(θ, 𝒮, τ(θ, 𝒮))
    

    

      
SymbolMeaning

      
dDiagnosis text — what went wrong and why

      
LMreflectA language model acting as a "critic"

      
θThe current prompt candidate

      
𝒮The minibatch of failed examples

      
τ(θ, 𝒮)Execution trace — the full input/reasoning/output chain

    

    
Step 2 — Mutation

    
The Mutate operator takes the original prompt and the diagnosis and produces an improved prompt:

    

      Mutate
      θ' = LMmutate(θ, d)
    

    

      
SymbolMeaning

      
θ'The new, (hopefully) improved prompt candidate

      
LMmutateA language model acting as a "fixer" / "editor"

      
dThe diagnosis from the Reflect step

    

    
Together, these form a full transformation: θ → τ → d → θ'

      →
      
Run on 𝒮
get traces τ

      →

diagnosis d

      →
      
LM_mutate
new prompt θ'

      →
      
Accept if F(θ') ≻ F(θ)

Selection from the Pareto Front

    

    
After mutation, GEPA has multiple candidate prompts. How does it decide which ones to keep? It maintains a Pareto Archive — only Pareto-optimal candidates survive.

    

      
Why keep multiple candidates?

      Different use cases need different trade-offs. A research assistant might want maximum accuracy.
      A customer service bot needs conciseness. GEPA's Pareto archive gives you the entire frontier
      of trade-offs, and you pick the point that fits your deployment.
    

    
The Archive Update Rule

    

      Archive update
      𝒜t+1 = { θ ∈ 𝒜t ∪ {θ'} : ¬∃φ ∈ 𝒜t ∪ {θ'} s.t. φ ≻ θ }
    

    

      
Plain English

      The new archive is: take the old archive plus the new candidate, then throw away anything that
      is dominated by something else. Only the "nobody beats me on everything" prompts survive.

Actionable Side Information vs Scalar Rewards

    

    
This is the key insight that makes GEPA 35x more sample-efficient than RL. Let's compare what each optimizer "sees."

    

      

        
RL / GRPO sees

        Score: 0.62


        That's it. One number. No idea what went wrong.


        To improve, you must randomly sample thousands of prompts until something works.
      

      

        
GEPA sees (ASI)

        Score: 0.62 + trace of every failure + natural-language diagnosis


        Knows exactly what failed and why. Makes a targeted one-step fix.
      

    

    

      
Information toy model

      
Suppose a score ∈ [0,1] discretized to 0.01 precision. That gives ~7 bits of information per evaluation.

      
Actionable Side Information (full trace + diagnosis) might contain thousands of tokens of targeted feedback — orders of magnitude more information per sample.

      
This is why GEPA achieves the same optimization quality in ~35x fewer LLM evaluations compared to GRPO.

The full algorithm in pseudocode

      
# GEPA algorithm
def gepa_optimize(program, dataset, objectives, max_iters):
    archive = bootstrap(program, dataset)  # run DSPy's Bootstrap first
    for t in range(max_iters):
        theta = sample_from_pareto_front(archive)
        S = sample_minibatch(dataset)
        tau = run_with_traces(theta, S)
        diagnosis = LM_reflect(theta, S, tau)
        theta_new = LM_mutate(theta, diagnosis)
        scores_new = evaluate(theta_new, objectives)
        archive = update_pareto_archive(archive, theta_new, scores_new)
    return archive

From Math to Code

    

    
Here's how the math translates to actual DSPy code. Expand each part to see the connection.

    

      

        Define your objectives (the Fi functions)
        ▼
      

      

        
Each objective is a Python function that returns a score in [0,1]:

        
# Objective 1: Accuracy
def accuracy(example, pred, trace=None):
    return float(pred.answer.strip() == example.answer.strip())

# Objective 2: Conciseness (shorter is better, up to 200 chars)
def conciseness(example, pred, trace=None):
    length = len(pred.answer)
    return max(0, 1 - length / 200)

# These are F₁ and F₂ from the math

      

    

    

      

        Run GEPA (the full algorithm loop)
        ▼
      

      

        
import dspy
lm = dspy.LM("openai/gpt-4o-mini")
dspy.configure(lm=lm)

program = dspy.ChainOfThought("question -> answer")

optimizer = dspy.GEPA(
    metric=[accuracy, conciseness],  # F(θ) = [F₁, F₂]
    max_iters=10,
)
optimized = optimizer.compile(program, trainset=trainset)

best_accurate = optimized.pareto_front[0]   # most accurate
best_concise  = optimized.pareto_front[-1]   # most concise
