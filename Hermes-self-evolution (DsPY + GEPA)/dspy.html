The Problem DSPy Solves

Analogy first Imagine writing a web application where you had to manually write every CPU instruction. That would be insane — you use a compiler. Building LLM apps without DSPy is the same: you are manually "compiling" prompts by hand, and every time you change the AI model, they break.

The Three Core Abstractions

Every DSPy program is built from exactly three things. Click each block to expand:

📋 Signature — The Contract

A Signature declares what inputs go in and what outputs should come out without specifying how to get there. Think of it as a typed function signature for AI tasks.

String form (quickest)

# "inputs -> outputs"
qa = dspy.Predict("question -> answer")
result = qa(question="What is DSPy?")
print(result.answer)

Class form (full control)

class SummarizeCode(dspy.Signature):
    """Summarize what this code does."""
    code: str = dspy.InputField()
    summary: str = dspy.OutputField(
        desc="2-3 sentence summary"
    )
How the optimizer sees it The docstring becomes the task instruction. The desc guides output content. The optimizer can rewrite both — that is how it improves prompts.

⚙️ Module — The Strategy

A Module wraps a Signature and adds a prompting strategy. Two modules can share the exact same Signature but prompt the LLM differently.

ModuleStrategyUse when…
PredictDirect answerSimple fast tasks, classification
ChainOfThoughtReason step-by-step, then answerAny reasoning or multi-hop problem
ReActReason + call tools in a loopAgentic tasks requiring tool use
ProgramOfThoughtWrite & execute Python codeMath, computation, data analysis
MultiChainComparisonGenerate N candidates, compareHigh-stakes decisions
RetryRe-call with error feedbackWhen output must satisfy constraints
# Same Signature, three different strategies:
direct  = dspy.Predict("question -> answer")
thinker = dspy.ChainOfThought("question -> answer")
agent   = dspy.ReAct("question -> answer", tools=[search_web])

🔧 Optimizer — The Compiler

The Optimizer takes your program + training examples + a scoring function and automatically finds better prompts. No gradients, no GPU.

Bootstrap — Run your unoptimized program on training examples. Record every LLM call (the "trace").
Generate Candidates — Extract successful traces as few-shot examples. Ask an LLM to propose improved instruction variants.
Evaluate — Test each instruction+demo combo on a validation set. Score with your metric function (0.0–1.0).
Save Winner — Serialize the best combination as a .json file. Load it in milliseconds later.

Building Your First DSPy Program

import dspy

# Step 1: Connect to an LLM
lm = dspy.LM("anthropic/claude-sonnet-4-20250514")
dspy.configure(lm=lm)

# Step 2: Declare WHAT you want (Signature)
class AnswerWithReasoning(dspy.Signature):
    """Answer the question based on the context. Cite sources using [1], [2] notation."""
    question: str = dspy.InputField(desc="The user's question")
    context:  str = dspy.InputField(desc="Background information")
    answer:   str = dspy.OutputField(desc="Precise answer with inline citations")

# Step 3: Define HOW to execute (Module)
class SmartQA(dspy.Module):
    def __init__(self):
        self.answer = dspy.ChainOfThought(AnswerWithReasoning)
    def forward(self, question, context):
        return self.answer(question=question, context=context)

# Step 4: Run it immediately (no optimization required)
qa = SmartQA()
result = qa(question="What is gradient descent?", context="Gradient descent minimizes...")
print(result.rationale)  # step-by-step reasoning
print(result.answer)

# Step 5: Optimize against your own examples
def my_metric(example, prediction, trace=None):
    return score_answer_quality(prediction.answer, example.expected_answer)

optimizer = dspy.MIPROv2(metric=my_metric, auto="medium")
compiled  = optimizer.compile(qa, trainset=your_training_data)
compiled.save("optimized_qa.json")

# Later: load without re-optimizing
qa2 = SmartQA()
qa2.load("optimized_qa.json")

Composing a Multi-Step Pipeline

class ResearchPipeline(dspy.Module):
    def __init__(self):
        self.find_papers  = dspy.ReAct("topic -> papers: list", tools=[search_arxiv])
        self.synthesize   = dspy.ChainOfThought("topic, papers -> synthesis, gaps")
        self.write_review = dspy.Predict("topic, synthesis, gaps -> review: str")
    def forward(self, topic):
        papers    = self.find_papers(topic=topic)
        synthesis = self.synthesize(topic=topic, papers=papers.papers)
        review    = self.write_review(topic=topic, synthesis=synthesis.synthesis, gaps=synthesis.gaps)
        return review

# The optimizer improves ALL three sub-modules simultaneously
optimizer = dspy.GEPA(metric=review_quality_metric)
compiled  = optimizer.compile(ResearchPipeline(), trainset=data)

Integrating DSPy with LangGraph

LangGraph owns the workflow; DSPy owns LLM call quality within each step.

from langgraph.graph import StateGraph

import dspy

lit_reviewer = LitReviewPipeline()

class PaperState(TypedDict):
    topic: str; literature: str

def lit_review_node(state: PaperState) -> PaperState:
    result = lit_reviewer(topic=state["topic"])

workflow = StateGraph(PaperState)

workflow.set_entry_point("literature")
app = workflow.compile()

Q: What is the relationship between a Signature and a Module?

Correct! Signature = WHAT (typed inputs/outputs). Module = HOW (Predict, ChainOfThought, ReAct…). You can swap the Module without touching the Signature.

What Even Is an LLM? (2 minutes)

Before anything else — let's make sure we understand what we're working with.

🔮
The World's Best Autocomplete

Your phone keyboard suggests the next word when you type. An LLM (Large Language Model) is the same idea, but it has read most of the text ever written by humans, so its suggestions are extraordinarily good.

It doesn't "think" or "understand" — it calculates: "given everything before this point, what text is most likely to come next?"

⚠️ The LLM has no idea what YOU want. It just produces the statistically likely continuation. To get useful output, you must craft careful instructions — that's the "prompt."

So What Is a Prompt?

A prompt is the text you give the LLM to steer what it generates.

Vague prompt

"Translate this text: Bonjour"

Output: "Translate this text: Hello" — it continued the pattern, not your intent.

Clear prompt

"Translate the French word below into English. Return ONLY the English word.

French: Bonjour
English:"

Output: "Hello" ✓

The DSPy Solution: Separate WHAT from HOW

This is the core insight. Everything else follows from it.

Key Concept: "Declarative"

When you say WHAT you want without specifying HOW to get there, that's called declarative. The opposite — specifying every step — is called imperative.

✅ Imperative vs Declarative — Click to Compare

You specify every instruction manually:

# OLD WAY
prompt = """You are an expert AI assistant. When given a research
question and context, provide a thorough, precise answer.
Structure your response: first acknowledge the question,
then analyze the context, then provide your answer. Cite
sources inline using [1] notation. Do not exceed 200 words.
Question: {question}
Context: {context}
Answer:"""
❌ Breaks on new models. Hard to improve. 20 hours of tuning for one task.

You specify only the contract — what goes in and what comes out:

# DSPY WAY
class AnswerQuestion(dspy.Signature):
    """Answer the question using the context. Cite sources."""
    question: str = dspy.InputField()
    context:  str = dspy.InputField()
    answer:   str = dspy.OutputField()
✓ DSPy generates the prompt automatically. Switch models? Re-optimize automatically.

Part 5 · Building Block #2

Modules — The Strategy

A Module takes your Signature and decides how to execute it.

👨‍🍳
Same Recipe, Different Chef Techniques

A recipe (Signature) says: "inputs: flour, eggs → output: cake." But how you make that cake depends on your technique:

  • Predict: Follow the recipe literally, answer immediately.
  • ChainOfThought: Think through the steps before acting.
  • ReAct: Reason + use tools (oven, mixer, etc.) in a loop.
⚙️ See the Modules in Action

Task: "What is 17 × 8?"

Code:

predictor = dspy.Predict("question -> answer")
result = predictor(question="17 × 8?")
LLM output:

Input: "17 × 8?"
Output: "142" ✗

No thinking. Often wrong on math.
Predict — When to use it

Simple tasks where the answer is obvious. Classification, data extraction. Fast and cheap but no reasoning.

Code:

cot = dspy.ChainOfThought("question -> answer")
result = cot(question="17 × 8?")
LLM output:

Reasoning: "17×8 = 17×(4+4) = 68+68 = 136"
Output: "136" ✓
ChainOfThought — When to use it

Any task requiring reasoning, multi-step logic. Use for math, argument analysis, complex Q&A.

Code:

react = dspy.ReAct("topic -> summary",
  tools=[search_arxiv])
result = react(topic="GEPA paper")
LLM output:

Thought: "I need to search"
Action: search_arxiv("GEPA")
Obs: [results...]
Output: "GEPA is..."
ReAct — When to use it

Tasks requiring external tools: web search, database queries, API calls, file reading.

The Math of Chain of Thought

Without reasoning, the AI tries to go directly from input to output:

P( answer | question )
"Probability of getting the right answer, given only the question"

With Chain of Thought, we split the problem. We first generate a reasoning chain r, then the answer:

P( answer | question ) = ∑r P( answer | r, question ) × P( r | question )
"Sum over all reasoning chains r: P(correct answer given reasoning) × P(that reasoning)"
Part 6 · Building Block #3

Optimizers — The Automatic Improvement Machine

🏔️
Climbing a Mountain in the Dark

Imagine you're on a mountain range at night. You can't see anything, but you can feel whether you're going uphill or downhill. Optimization is the same: you can't "see" which prompt is best, but you can evaluate prompts and climb toward better ones.

What Does "Optimize" Mean Formally?

θ* = argmaxθ 𝔼(x,y)∼𝒟 [M(Pθ(x), y)]
θ=Our prompt settings (instruction text + example demonstrations). This is what gets optimized.
θ*=The best prompt settings — what we're trying to find (star means "optimal").
argmaxθ="Find the value of θ that makes everything after it as large as possible."
𝔼[...]=Expected value = average. We average over all our test examples.
(x,y)∼𝒟=Sampling input-output pairs from dataset 𝒟. x is a question, y is the expected answer.
M(a,b)=The metric function: how good was prediction a vs. correct answer b? Returns 0.0–1.0.
Pθ(x)=Our AI program with settings θ, applied to input x. Produces a prediction.
In Plain English

"Find the prompt settings θ that, on average across all our test examples, make our AI program produce outputs that score highest according to our metric."

Why Can't We Use Gradients?

Neural net weights (numbers)

Parameter: w = 0.7423
Gradient says: "increase w by 0.01"
New w: 0.7523 → score improves ✓

Works because numbers are continuous.

Prompt text (discrete)

Parameter: "Answer concisely."
Gradient says: "increase θ by 0.01"
New θ: "Answer concisely.???" ← meaningless ✗

Text is discrete — can't "add 0.01" to a word.

Quick Questions

Q1: What is "declarative" programming?

Correct! Declarative = describe the goal. Imperative = describe the steps. DSPy is declarative: you say "I want (question, context) → answer" and DSPy figures out how to prompt the LLM.

Q2: In θ* = argmax 𝔼[M(P_θ(x), y)], what does M represent?

Correct! M is your metric — the function that takes (prediction, expected_answer) and returns a score 0.0–1.0.

Q3: Why does Chain of Thought outperform direct Predict on reasoning tasks?

Correct! CoT computes P(y|r,x)·P(r|x) summed over reasoning chains r. Each factor is easier to estimate than P(y|x) directly.
