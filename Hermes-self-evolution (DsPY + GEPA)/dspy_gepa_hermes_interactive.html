DSPy — The AI Compiler

Stop writing prompts by hand. DSPy lets you declare what you want from an AI and automatically finds the best prompt to get it.

Python library·Works with Claude, GPT, Gemini·No GPU needed

The Problem DSPy Solves

Analogy first Imagine writing a web application where you had to manually write every CPU instruction. That would be insane — you use a compiler. Building LLM apps without DSPy is the same: you are manually "compiling" prompts by hand, and every time you change the AI model, they break.

Without DSPy

Fragile. Hard-coded. Breaks when you change models. Impossible to optimize systematically.

With DSPy

Declarative. Model-agnostic. The prompt text is generated and optimized automatically.

The Three Core Abstractions

Every DSPy program is built from exactly three things. Click each block to learn what it does:

📋 Signature — The Contract

A Signature declares what goes in (inputs) and what should come out (outputs) from an AI call — without saying anything about how to get there. Think of it as a typed function signature, but for an AI.

String form (quickest)

# Minimal: "input -> output"
qa = dspy.Predict("question -> answer")
result = qa(question="What is DSPy?")
print(result.answer)

# With type hints
clf = dspy.Predict("text -> label: bool")

Class form (full power)

class SummarizeCode(dspy.Signature):
    """Summarize what this code does
    in plain English."""
    code:    str = dspy.InputField()
    summary: str = dspy.OutputField(
        desc="2-3 sentence summary"
    )
Key Insight The docstring becomes the instruction to the AI. The optimizer can rewrite both the docstring and the field descriptions to find the phrasing that works best. That is how automatic optimization works.

⚙️ Module — The Strategy

A Module wraps a Signature and adds a prompting strategy — the technique used to coax the AI into giving the best answer. Two modules can share the same Signature but prompt the AI in completely different ways.

ModuleStrategyUse when…
PredictDirect answer, no extrasSimple, fast tasks
ChainOfThoughtThink step-by-step, then answerReasoning problems
ReActReason + call tools in a loopAgentic tool-use
ProgramOfThoughtWrite & execute code to get answerMath, computation
MultiChainComparisonTry N approaches, pick bestHigh-stakes decisions
Same Contract, Different Strategy You can swap Predict for ChainOfThought at any time without changing your Signature. The "what" stays fixed; only the "how" changes. This separation is what makes DSPy programs easy to improve.

🔧 Optimizer — The Compiler

The Optimizer takes your program + training examples + a scoring function and automatically searches for better prompts. No gradients. No GPU. Just discrete search over text.

Bootstrap — Run your unoptimized program on training examples. Collect execution traces (every AI call + its output).
Generate Candidates — Extract successful traces as few-shot examples. Ask another LLM to propose improved instruction variants.
Evaluate — Test each instruction+demo combination on a held-out validation set. Score with your metric function.
Select Winner — Save the best combination as a .json file. Load it later with zero re-optimization.
No GPU, No Gradients Everything is text search evaluated by running the program and scoring the output. Works with closed-source APIs (Claude, GPT, Gemini). The compiled .json is just the winning prompt text — extract it and use it anywhere.

Building Your First DSPy Program

Here is a complete working example with every step explained:

import dspy

# ─── Step 1: Connect to an LLM ────────────────────────────────────────
lm = dspy.LM("anthropic/claude-sonnet-4-20250514")
dspy.configure(lm=lm)

# ─── Step 2: Declare WHAT you want (Signature) ───────────────────────
class AnswerWithReasoning(dspy.Signature):
    """Answer the question based on the provided context.
    Cite sources using [1], [2] notation."""
    question: str = dspy.InputField(desc="The user's question")
    context:  str = dspy.InputField(desc="Background information")
    answer:   str = dspy.OutputField(desc="Precise answer with citations")

# ─── Step 3: Define HOW to execute (Module) ──────────────────────────
class SmartQA(dspy.Module):
    def __init__(self):
        # ChainOfThought adds a "rationale" step before the answer
        self.answer = dspy.ChainOfThought(AnswerWithReasoning)

    def forward(self, question, context):
        return self.answer(question=question, context=context)

# ─── Step 4: Run it immediately (no optimization required yet) ────────
qa = SmartQA()
result = qa(
    question="What is gradient descent?",
    context="Gradient descent minimizes a function by moving against the gradient..."
)
print(result.rationale)  # The reasoning trace (added by ChainOfThought)
print(result.answer)     # The final answer

# ─── Step 5: Optimize it against your own examples ───────────────────
def my_metric(example, prediction, trace=None):
    # Return 0.0-1.0: how good was this prediction?
    return score_answer(prediction.answer, example.expected_answer)

optimizer = dspy.MIPROv2(metric=my_metric, auto="medium")
compiled  = optimizer.compile(qa, trainset=your_training_data)
compiled.save("optimized_qa.json")   # Reuse forever, no re-optimizing

# ─── Later: load without any re-optimization ─────────────────────────
qa2 = SmartQA()
qa2.load("optimized_qa.json")         # Instant, cheap to run

Composing Multiple Modules into a Pipeline

The real power of DSPy is that you can chain modules in plain Python — any if, for, or function call is fine. The structure is implicit in your Python code.

class ResearchPipeline(dspy.Module):
    def __init__(self):
        # Three sub-modules, each with its own signature
        self.find_papers  = dspy.ReAct("topic -> papers: list", tools=[search_arxiv])
        self.synthesize   = dspy.ChainOfThought("topic, papers -> synthesis, gaps")
        self.write_review = dspy.Predict("topic, synthesis, gaps -> review: str")

    def forward(self, topic):
        papers    = self.find_papers(topic=topic)
        synthesis = self.synthesize(topic=topic, papers=papers.papers)
        review    = self.write_review(
            topic=topic,
            synthesis=synthesis.synthesis,
            gaps=synthesis.gaps
        )
        return review

# Compile the whole pipeline at once
# The optimizer improves ALL three sub-modules simultaneously
optimizer = dspy.GEPA(metric=review_quality_metric)  # See GEPA tab!
compiled  = optimizer.compile(ResearchPipeline(), trainset=data)

Which Optimizer Should You Use?

OptimizerStrategyBest ForEvaluations Needed
BootstrapFewShot Collect successful traces as examples Quick baseline, small datasets ~50–200
MIPROv2 Bayesian search over instruction+example combos General purpose, single-module ~200–1000
GEPA ✨ Reflective evolutionary search (see GEPA tab) Complex pipelines, agents, tool use 100–500
GRPO Gradient-based RL, fine-tunes model weights When you have a GPU 5,000–25,000+
When to choose GEPA over MIPROv2 If your program has multiple steps, uses tools, or fails in complex ways that are hard to score with a single number — use GEPA. It can read the full execution trace and diagnose exactly where the failure occurred, then propose a targeted fix.

Check Your Understanding

Q: You want to build a system that searches the web for papers and summarizes them. Which DSPy module would you use for the search step?

Correct! ReAct (Reason + Act) is the module designed for tool-using agents. It lets the AI call your search function and reason about the results in a loop until it has enough information. The other modules don't support tool calls.

Q: After running the optimizer, what artifact does it produce that you can save and reuse?

Correct! The compiled output is just a .json containing the winning instruction text and few-shot examples. No GPU, no model weights. You can load it instantly any time without re-running the optimizer.

GEPA — The Evolutionary Optimizer

Evolution for prompts. GEPA uses an AI to read failure logs, understand why something went wrong, and propose a targeted fix. Then it repeats.

35x cheaper than RL·No GPU·Human-readable logs

Why Not Just Use Reinforcement Learning?

Analogy Imagine teaching someone to bake by only ever saying "that scored 0.4" after each attempt. That is Reinforcement Learning. GEPA instead says: "The cake was good but you used too much salt in step 2, and you forgot to preheat the oven." Now the learner knows exactly what to fix.

Reinforcement Learning (RL)

  • Returns a scalar reward: "0.43"
  • Needs 5,000–25,000+ evaluations to converge
  • Requires GPU + direct model weight access
  • Cannot explain why something failed
  • Takes hours to days to run
  • Expensive compute infrastructure

GEPA

  • Reads full execution traces: "step 3 failed because…"
  • Needs only 100–500 evaluations
  • API-only, no GPU required
  • Human-readable improvement logs
  • Takes minutes to a few hours
  • Costs ~$2–10 per optimization run

📊 Evaluate

GEPA runs each candidate prompt on your test examples. For every example it captures three things, not just a score:

  • Score — How well the output matched expectations (0.0–1.0)
  • Execution trace — Every step the AI took, including tool calls, reasoning, and intermediate outputs
  • Feedback string — Your natural-language description of what went right or wrong

This richness is the key difference from RL, which only sees the scalar score.

🔍 Reflect

A Reflection LLM (e.g., GPT-5 or Claude Opus) reads the execution traces from failed or low-scoring runs and produces a natural-language diagnosis:

# What GEPA sends to the Reflection LLM:
"""
Prompt: "Answer the question concisely."
Test case: "What is gradient descent?"
Expected: A clear mathematical explanation with the update rule
Actual: "Gradient descent finds the minimum of a function."
Feedback: "Too vague. Missing the update rule: theta -= lr * grad(L)"
Trace: [single AI call, no reasoning step, no equations]

Diagnose the failure. What specifically needs to change in the prompt?
"""

Reflection LLM responds: "The prompt does not request mathematical detail. It needs to specify that technical answers should include the update rule and relevant notation."

✎️ Mutate

Based on the reflection, GEPA generates N new prompt candidates that address the diagnosed issue. These are semantic mutations guided by understanding — not random noise.

# Mutation examples:
original:   "Answer the question concisely."

mutation_1: "Answer the question. For algorithm or math questions,
             include the formal update rule or equation."

mutation_2: "Provide a precise technical answer with: (1) an intuitive
             explanation, (2) the mathematical formulation, (3) key
             parameters. Use LaTeX notation for equations."
Evolutionary Memory: Each mutation also inherits lessons from all ancestral candidates in the search tree — so improvements compound across generations.

🏆 Select — The Pareto Front

New candidates are evaluated, then added to the Pareto Front if they are "non-dominated." A candidate is non-dominated when no other candidate beats it on all objectives at once.

This gives you a menu of candidates at different quality/cost trade-offs, rather than collapsing to a single winner:

The dominated point (gray) is beaten by the "Balanced" point on both quality and cost — so it is discarded. You end up with a menu of non-dominated options.

The Feedback Function — Your Most Important Design Choice

The Golden Rule of GEPA The richer your feedback string, the better the mutations. A score of "0.6" produces vague mutations. A string like "citation format was wrong in step 2: expected [1] but got (Smith, 2024)" produces a precise, surgical fix. This is why GEPA beats RL. 
def my_feedback(gold, pred, trace=None) -> tuple[float, str]:
    """
    Returns: (score 0.0-1.0,  feedback_string)

    The feedback string is NATURAL LANGUAGE.
    It describes WHAT went wrong in plain English.
    This is what the reflection LLM reads to propose the next mutation.
    """
    score  = 0.0
    issues = []

    # Check 1: citation format
    if pred.citation_format_correct():
        score += 0.5
    else:
        issues.append(
            f"Citation format wrong. Expected [1] Smith (2024), "
            f"got '{pred.citations[:40]}'"
        )

    # Check 2: word count
    if 200 <= pred.word_count() <= 300:
        score += 0.3
    else:
        issues.append(
            f"Word count {pred.word_count()} outside target range 200-300."
        )

    # Check 3: passive voice (for academic writing)
    if pred.passive_voice_ratio() > 0.6:
        score += 0.2
    else:
        issues.append("Should use passive voice throughout (academic register).")

    # This rich string is the key input to GEPA's reflection step
    feedback = " | ".join(issues) if issues else "Looks good!"
    return score, feedback

      Phase 2
      Tool Description Evolution
      Planned
    

    

      
Target: The short string that tells the AI what each tool does and when to call it.

      
Why it matters: Tool descriptions affect every single tool call across the entire session. A better description means the AI calls tools more accurately and with better arguments.

      

        
Before: "search_papers: Search for academic papers."

        
After GEPA: "search_papers: Search Semantic Scholar by topic, author, or keyword. Returns titles, abstracts, DOIs. Use for prior work. Do NOT use for full PDF text (use fetch_pdf for that)."

      

    

  


  

    

    

      Phase 3
      System Prompt Evolution
      Planned
    

    

      
Target: The system prompt sections that define Hermes's overall behavior, persona, and priorities.

      
Why it is harder: System prompts affect everything. A bad mutation cascades across all tasks. Requires fine-grained section-by-section optimization with targeted test suites to avoid unintended regressions.

      
Approach: Divide the system prompt into named sections. Evolve each section independently against a test suite specific to that section's purpose.

    

  


  

    

    

      Phase 4
      Code Evolution (Darwinian Evolver)
      Planned
    

    

      
Target: The actual Python implementation of tools — not just their descriptions, but their runtime behavior.

      
Different engine: Uses the imbue-ai/darwinian_evolver instead of GEPA, because code has hard correctness constraints (it must compile, pass tests, run without crashing). GEPA's text-mutation approach would too easily produce broken code.

      
How it works: Each candidate is an actual git commit. Mutations are AI-generated code edits. Selection is based on test pass rate + performance benchmarks. Crossover combines sections from two high-performing commits. Think genetic algorithms, but operating on git history.

    

  


  

    

    

      Phase 5
      Continuous Improvement Loop
      Planned
    

    

      
Target: A fully automated pipeline that runs all phases on a schedule, reads your session history, identifies skill failures, and deploys improvements without manual intervention.

      
The vision: A nightly or weekly cron job that reads your real conversations, finds which skill files contributed to poor outcomes, runs GEPA to improve those specific skills, passes guardrail checks, and deploys the winners. Your AI agent gets marginally better every week, automatically, at the cost of a cup of coffee per run.

    

  









How Phase 1 Works Step by Step




  

    Input
    
A Hermes skill file, e.g. github-code-review.md. Contains: name, description, step-by-step procedure, examples, edge cases.

  

  

    Build Evaluation Dataset
    
Synthetic: An LLM generates 20 realistic tasks that should trigger this skill, plus expected outcomes. Fast, always available.
SessionDB: Reads ~/.claude/history.jsonl and your Copilot sessions. Filters for conversations where this skill would have been relevant. Uses your real work patterns.

  

  

    Define the Metric
    
Run Hermes with the current skill on each test case. Score the result (task completion rate, output quality). Generate a rich feedback string: "The skill failed because it didn't specify to check test coverage before approving."

  

  

    GEPA Evolution Loop
    
The skill file itself is the "prompt" being optimized. GEPA runs 10–15 iterations: evaluate → reflect → mutate → select. The Pareto front balances skill quality against skill file size.

  

  

    Guardrail Checks
    
Every evolved variant must pass: pytest tests/ -q (100%), size ≤ 15KB, caching compatibility, and a semantic preservation check (LLM judge: "Does this still do what the original claimed?"). This prevents GEPA from gaming the metric by changing the skill's purpose.

  

  

    Output: Deploy or PR
    
Either opens a PR against the Hermes repo for human review, or you copy directly to ~/.hermes/skills/ for immediate local use. The PR model is conservative and recommended for shared environments.

  









SessionDB — Learning From Your Real Work




  Key Insight
  Every conversation where Claude Code didn't quite do what you wanted is a signal that a skill file could be better. SessionDB harvests that signal automatically from your real session history.




The pipeline reads from these files on your machine:


~/.claude/history.jsonl          # Your Claude Code sessions
~/.hermes/sessions/              # Hermes's own session store
~/.copilot/session-state/*/      # GitHub Copilot sessions



It then filters with a two-stage pipeline:




  

    
Stage 1: Keyword Filter

    
Scan each conversation for keywords related to the skill. Very fast — runs in milliseconds. Discards obvious mismatches cheaply.

    
Example: discard conversations about Python dependencies when evolving the "latex-writing" skill.

  

  

    
Stage 2: LLM-as-Judge

    
For conversations that pass stage 1, ask an LLM: "Would the latex-writing skill have been relevant here? Score 0–1." Built as a DSPy module, so it too can be optimized.

    
Keep conversations with judge score > threshold. These become your eval set.

6 Mental Models to Remember




  

    
1. Prompts are compiled programs

    
A prompt is not just text — it is a program encoding your intent. Like all programs, it can be compiled (DSPy), tested (metric function), and optimized (GEPA). Stop writing prompts by hand.

  

  

    
2. Scalars are lossy

    
Reducing a failure to a score of "0.4" throws away the diagnosis. GEPA preserves the full execution trace so a reflection LLM can understand exactly what went wrong and propose a precise fix.

  

  

    
3. Targeted > Random

    
Random prompt variations are inefficient. A targeted mutation — "fix the citation format reference in step 2" — converges 35x faster than RL-style random perturbation.

  

  

    
4. Skills are textual programs

    
A markdown skill file has implicit inputs and outputs and measurable behavior. It can be evaluated on benchmark tasks and improved via GEPA. This insight powers Hermes self-evolution.

  

  

    
5. Your history is training data
