BO-Tuned Cache Eviction

Tutorial 1/8 · Problem Launch · Light (~1.5 hrs)

Production caches use hand-tuned eviction policies like LRU, LFU, or ARC. But every workload is different. What if a Bayesian optimizer could learn the optimal eviction strategy from request traces — and adapt as the workload shifts?

The Problem

A cache has finite capacity. When a new item arrives and the cache is full, something must be evicted. The eviction policy decides what.

Hand-tuned policies (LRU, LFU, ARC) make assumptions about the workload: LRU assumes recent access predicts future access, LFU assumes frequency predicts future access. But real workloads are non-stationary — access patterns shift throughout the day.

Can we frame cache eviction as a Bayesian optimization problem? Treat the cache's hit rate as a black-box function f(policy_params, workload_slice) and search for the policy that maximizes it under shifting workloads.

Eviction policies answer: "Who gets kicked out?" ┌─ Policy Comparison ───────────────────────────────────────┐ │ │ │ LRU: Evict the oldest-accessed entry │ │ Assumes: recently used → will be reused │ │ Works for: recency-heavy workloads │ │ Fails when: a burst of one-time requests │ │ flushes the hot cache │ │ │ │ LFU: Evict the least-frequently accessed │ │ Assumes: frequency → future importance │ │ Works for: stable popularity distributions │ │ Fails when: a once-popular item becomes irrelevant │ │ but has high count │ │ │ │ ARC: Adaptive — switches between LRU/LFU │ │ based on recent workload patterns │ │ Better than either alone, but still │ │ uses heuristics, not optimization │ │ │ │ BO-Policy: Learn from trace data (see next stage) │ └───────────────────────────────────────────────────────────┘
The BO approach frames eviction as optimization: ┌─ BO-Tuned Policy ──────────────────────────────────────────┐ │ │ │ Objective: maximize hit_rate(θ, trace) │ │ │ │ Parameters θ = [α, β, γ, window_size, ...] │ │ α: weight given to recency │ │ β: weight given to frequency │ │ γ: weight given to entry size │ │ window_size: lookback window for stats │ │ │ │ Score(key) = α · recency(key) + β · frequency(key) │ │ - γ · size(key) │ │ │ │ Instead of hand-tuning α,β,γ: │ │ → GP surrogate learns the mapping f(θ) → hit_rate │ │ over a trace window │ │ → EI acquisition proposes next θ │ │ → After N evals, the best θ is deployed │ │ │ │ Warm-start from known-good defaults │ │ (ARC's parameters = starting point) │ └───────────────────────────────────────────────────────────┘
The beautiful part: workloads drift, and BO adapts. ┌─ Adaptive Loop ────────────────────────────────────────────┐ │ │ │ Timeline: │ │ │ │ 08:00 — Workload: batch processing jobs │ │ Best policy: LFU-heavy (α=0.2, β=0.7) │ │ │ │ 12:00 — Workload: interactive users │ │ Best policy: LRU-heavy (α=0.7, β=0.2) │ │ │ │ 18:00 — Workload: mixed (both) │ │ Best policy: balanced (α=0.4, β=0.4) │ │ │ │ Every N minutes: collect trace window, │ │ re-optimize θ on the recent window, │ │ deploy the new policy. │ │ │ │ BO's advantage over RL: sample efficiency. │ │ A GP needs ~20-50 evaluations to converge; │ │ RL needs thousands of online interactions. │ │ Less disruption to production traffic. │ └───────────────────────────────────────────────────────────┘
