Cache Eviction: Recency vs. Frequency

Tutorial 1/8 · The core tension · Light (~1.5 hrs)

A cache is a small, fast storage layer that holds a subset of data so future requests can be served faster. Since the cache is smaller than the backing store, it must evict items when it runs out of room. The eviction policy — which item to kick out — is the single most important design decision for cache performance.

Every eviction policy faces the same fundamental question using only two signals:

⏱️
Recency
When was this item
last accessed?
📊
Frequency
How often is this item
accessed overall?

This tutorial is about why those two signals lead to different policies, when each one fails, and how Bayesian optimization can learn to balance them automatically.

1. What is a Cache?

Imagine a library desk with room for 4 books (the cache). The library basement has thousands of books (the backing store). Fetching a book from the basement takes 5 minutes. Fetching from the desk takes 10 seconds.

If a patron requests a book that's on the desk, that's a hit — 10 seconds. If it's in the basement, that's a miss — 5 minutes. The goal of a cache is to maximize hits by keeping the most useful items on the desk.

Key insight: An eviction policy is a bet about the future. You keep items you think will be requested again, and evict items you think won't. Different policies make different bets.

2. A Concrete Trace

Let's work with a concrete sequence of requests (a trace). Throughout this tutorial, we'll run this trace through different policies and compare hit rates:

Trace: [ 'A', 'B', 'C', 'A', 'B', 'C', 'A', 'B', 'C', 'D', 'D', 'E', 'A', 'F', 'A', 'G', 'A', 'H', 'A', 'I' ] Capacity: 3

This trace has a mix of temporal locality (A, B, C repeat in bursts — like checking a dashboard every few seconds) and one-time items (E, F, G, H, I appear once — like loading an article you'll never revisit).

The capacity is 3 — only 3 items fit in the cache at once. Let's see how different policies fare.

3. LRU: The Recency Heuristic

The Idea

Least Recently Used (LRU) evicts the item that hasn't been accessed for the longest time. It bets that recently accessed items are likely to be accessed again — this is temporal locality, one of the most reliable patterns in real workloads.

Think of a stack of papers on your desk: you put each paper you use on top. When the stack gets too tall, you throw away the paper at the bottom — the one you haven't touched in the longest time. That's LRU.

The Implementation

The standard implementation uses a hash map (for O(1) lookup) plus a doubly-linked list (for O(1) movement). Here's a simpler version using a dictionary with access-order tracking:

# LRU: evict the oldest-accessed item def lru_hit_rate(trace, capacity): cache = {} # key -> access order counter hits = 0 tick = 0 for key in trace: if key in cache: hits += 1 cache[key] = tick # update recency else: if len(cache) >= capacity: # Find the key with smallest tick (oldest) oldest = min(cache, key=lambda k: cache[k]) del cache[oldest] cache[key] = tick tick += 1 return hits / len(trace)

Walkthrough — Step by Step

Let's run it. Each MISS-fill adds an item to the cache (cold start). Once full, MISS-evict kicks out the oldest. A HIT refreshes the recency.

When LRU Works

LRU shines when the workload has strong temporal locality — the same items are requested in bursts. In a web server serving a live dashboard that refreshes every 5 seconds with the same data, LRU keeps that data hot.

When LRU Fails: The Scan Problem

LRU's fatal weakness is a scan — a sequential pass through many distinct items. Each new item evicts the previously hot items from the cache, even though those hot items will be needed again right after the scan ends.

Scan failure example: If a cache holds 4 popular items (A, B, C, D) and a scan of 8 one-time items (E, F, G, H, I, J, K, L) passes through, LRU will evict A, B, C, D one by one and replace them with E–L. When the next request comes for A — miss! LRU just flushed your working set.

4. LFU: The Frequency Heuristic

The Idea

Least Frequently Used (LFU) evicts the item accessed the fewest times. It bets that frequently accessed items are valuable — if an item has been popular in the past, it will likely stay popular.

The Implementation

# LFU: evict the least-frequently-accessed item def lfu_hit_rate(trace, capacity): cache = {} # key -> frequency in cache freqs = {} # key -> global frequency hits = 0 for key in trace: freqs[key] = freqs.get(key, 0) + 1 if key in cache: hits += 1 cache[key] += 1 # increment cached freq else: if len(cache) >= capacity: # Find the key with smallest frequency lfu_item = min(cache, key=lambda k: cache[k]) del cache[lfu_item] cache[key] = freqs[key] # initial freq return hits / len(trace)

When LFU Beats LRU: Zipf Workloads

Many real access patterns follow a Zipf distribution: the top 20% of items account for 80% of requests. Think search engine queries, news article visits, or YouTube views. LFU naturally keeps this hot set alive because those items accumulate high frequencies.

Many natural access patterns (web pages, words in text, file accesses) follow Zipf-like distributions. This is LFU's natural habitat.

LFU wins. It keeps the popular items in cache even when they haven't been accessed in a few ticks, while LRU may temporarily evict a popular item for a less-popular one that happened to arrive recently.

When LFU Fails: Cache Pollution

LFU's weakness is frequency staleness. Once an item has built up a high access count, it stays in cache even after its popularity has died. This is called cache pollution — the cache is full of previously-hot items that no longer attract requests.

Consider: the morning news cycle (A,B,C,D are hot), then the afternoon cycle (E,F,G,H are hot). LFU still keeps A,B,C,D because they have counts of 10 each. New items E,F,G,H enter with count 1 and get immediately evicted. The cache is polluted with yesterday's news.

Recency has a natural decay mechanism — an unaccessed item ages out. Frequency has no such decay — an old high count stays high forever, making LFU slow to adapt to shifts.

5. The Fundamental Tension

The examples above reveal an irreducible trade-off:

✅ LRU strong when
  • Workload has temporal locality
  • Working set fits in cache
  • Access patterns shift over time
❌ LRU weak when
  • Scan workload flushes hot cache
  • One-time items dominate
✅ LFU strong when
  • Zipf-like popularity distribution
  • Item popularity is stable
  • Scans don't pollute (counts stay low)
❌ LFU weak when
  • Popularity shifts over time
  • Cache pollution from old popular items
The takeaway: No single policy is optimal for all workloads. The optimal policy depends on the character of the access pattern — and real workloads change their character over time.

6. Belady's Optimal: The Theoretical Ceiling

There's a theoretical upper bound: Belady's optimal algorithm (OPT). It evicts the item that will be used farthest in the future. This achieves the maximum possible hit rate for any given trace.

OPT is impossible in practice — it requires knowledge of future requests. But it serves as a benchmark: any practical policy is measured by how close it gets to OPT.

ℹ️ How close do LRU and LFU get to OPT? ▶

Research shows that on real traces, LRU typically achieves 70–90% of OPT's hit rate. ARC (Adaptive Replacement Cache, which we'll cover in Tutorial 2) gets closer by dynamically switching between LRU and LFU behavior.

The gap between a good practical policy and OPT is where Bayesian optimization enters — if we can learn the right balance of recency and frequency from the trace itself, we can close that gap.

7. Where Bayesian Optimization Enters

The Hybrid Score Function

Instead of picking LRU or LFU, we can combine both signals into a single score and evict the item with the lowest score:

Score(key) = α · recency_z(key) + β · frequency_z(key)

Where each signal is normalized to [0, 1]:

recency_z = 1 − (now − last_accessed) / window_size frequency_z = count(key) / max_count_in_cache

The parameters α and β control the trade-off:

  • α = 1, β = 0 → pure LRU (recency only)
  • α = 0, β = 1 → pure LFU (frequency only)
  • α = 0.5, β = 0.5 → equal weight to both

Why Hand-Tuning Fails

You could manually pick α and β for a given workload. But:

  • Workloads shift over time — the right α at 8 AM might be wrong at 8 PM
  • There are more knobs — window_size, min frequency threshold, entry-size weight
  • Each cache serves different traffic — user-profile cache behaves differently from KV-cache

BO's Role: Learn (α, β) From Trace Data

Bayesian optimization treats the cache's hit rate as a black-box function of the policy parameters:

f(α, β) = hit_rate over a recent trace window

Each evaluation: pick (α, β) → run the policy on the last N requests → measure hit rate. The GP surrogate learns the landscape of f, and an acquisition function (EI) proposes the next (α, β) to try. After ~20–50 evaluations, BO converges to the best parameters for this workload, right now.

This is the problem we'll solve over the next 6 tutorials. We'll build a cache simulator, implement the hybrid score policy, collect traces, and train a GP surrogate to find the optimal (α, β) that maximizes hit rate — adapting as the workload shifts.

✋ Sharp Question

Click to reveal — On a Friday evening, a social media platform's traffic shifts from recency-heavy (live feed scrolling) to frequency-heavy (top-posts browsing). What happens to the optimal (α, β) over a 2-hour window? Sketch the adaptation trajectory.

Solution sketch:

At 5 PM during feed scrolling, temporal locality is high — users refresh the same feed every 30 seconds. LRU dominates → α ≈ 0.8, β ≈ 0.2.

At 6 PM during top-posts browsing, the same small set of viral posts gets millions of views. LFU dominates → α ≈ 0.2, β ≈ 0.8.

The adaptation trajectory: BO re-optimizes periodically (every N requests). At 5 PM it finds a high-α optimum. At 5:15 PM, α starts dropping as the workload shifts. By 6 PM, β is dominant.

The challenge: lag. If the workload shifts faster than the re-optimization window, the cache is stuck with stale parameters. One solution: keep multiple GP models at different timescales and ensemble them. Another: use the GP's uncertainty to detect regime changes (when prediction error spikes, re-optimize immediately).
